# kev/ — cross-vendor KEV/EPSS tracker

Generator for the **Cross-vendor KEV/EPSS** section published at `docs/kev/`. It records and
presents facts from public sources; it does **not** assess the predictive validity of EPSS or
KEV.

Migrated here from the private `jyz002xyz/kev_cross_vendor` prototype (kept as history, not
deleted). Moving it into this public repo removes the manual copy step and the need for a
cross-repo token: the generator, its tests, and the frozen snapshots now live in one place and
can regenerate `docs/kev/` directly.

## What it does
- Population = the **CISA KEV catalog** (fetched whole; keeps vendor/product/dates/ransomware
  flag/description).
- Window = **calendar month** (`dateAdded` in `YYYY-MM`).
- EPSS = **window CVEs only**, via the per-CVE FIRST API. Observed at first sighting and kept;
  historical EPSS is never reconstructed.
- NVD **publication date** is filled per CVE (stable past fact), labelled as NVD-publication,
  not vendor disclosure.
- Monthly snapshots are **frozen (gzip) and immutable** in `snapshots/{month}.json.gz`; the
  current month is an **open** window `snapshots/{month}.open.json.gz`, regenerated each run.
- **Backfill** past windows from the current catalog's `dateAdded`; their EPSS column is blank.

## Layout
    kev/
      kevtrack.py        # fetch + window/seal model + storage
      report.py          # internal-preview markdown/html
      publish.py         # public bilingual site (docs/kev/ layout)
      run.py             # orchestration: open current month, seal past months
      test_kevtrack.py   # offline tests
      snapshots/*.json.gz # frozen monthly data (immutable) + current open month
    docs/kev/            # published output (regenerated by run.py; NOT under kev/)

## Run
    python3 kev/run.py                 # regenerate docs/kev/ (fetches KEV/EPSS/NVD live)
    python3 kev/run.py --months 6
    python3 kev/test_kevtrack.py       # offline lifecycle tests
    python3 kev/test_integrity.py      # offline integrity-gate tests

Committing the regenerated `docs/kev/` is a **separate, gated step**: the daily workflow
`.github/workflows/kev-daily.yml` does it. Data updates only — code/display changes stay
behind normal PR review.

## Integrity gate (`integrity.py`)
Before generating or sealing, `run.py` fail-halts (non-zero exit, nothing written) if the KEV
catalog looks degraded — so a partial/broken fetch never publishes bad data and, critically,
never seals a month permanently incomplete (`kevtrack.seal()` never overwrites an existing
seal). Checks, all evaluated every run: fetch succeeded and non-empty; total count above a
floor and not dropped past a fraction vs the last successful run (KEV is ~monotonic); required
fields (`cveID`/`vendorProject`/`product`/`dateAdded`) present above a threshold; any month
about to be sealed has a non-empty window. Thresholds live in `integrity.py`, overridable via
env (`KEV_MIN_CATALOG`, `KEV_MAX_DECREASE_FRAC`, `KEV_MAX_MISSING_FRAC`, `KEV_ALLOW_EMPTY_SEAL`).
The last successful catalog size is recorded in `snapshots/catalog_meta.json` (git-tracked).

## Daily workflow (`.github/workflows/kev-daily.yml`)
`workflow_dispatch` + `schedule` (schedule **disabled until explicit owner go**). Runs
`python kev/run.py`, then: enforces the diff is confined to `docs/kev/` and `kev/snapshots/`;
skips the commit when `docs/kev/` is unchanged (no-op — `generated_at` is held stable so
identical data produces no diff); otherwise commits with a bot identity via `GITHUB_TOKEN`
(`contents: write`, no PAT). A per-run success ping to an external monitor (URL in the
`KEV_HEALTHCHECK_URL` secret, never committed) acts as a dead-man's switch.

## Dependencies
Python 3.10+ (uses `X | None` runtime types). **Standard library only** — no third-party
packages. Network endpoints:
- CISA KEV catalog — `https://www.cisa.gov/sites/default/files/feeds/known_exploited_vulnerabilities.json`
- FIRST EPSS API — `https://api.first.org/data/v1/epss`
- NVD CVE API 2.0 — `https://services.nvd.nist.gov/rest/json/cves/2.0` (no API key; respects 5 req/30s)

## Stance: descriptive, not evaluative
The report **records and presents facts**; it does not judge the predictive correctness of EPSS
or KEV. EPSS is shown as an attribute recorded at the time of KEV addition. KEV is not a
complete record of exploitation — it is what CISA confirmed and prioritized for federal
agencies. Vendor counts reflect federal deployment + CISA visibility, **not** security quality;
vendors are **never** ranked by count. Facts are machine-generated; interpretation is human.
Sealed snapshots are immutable. Bilingual order is English-first per the site convention.
